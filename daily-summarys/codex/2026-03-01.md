# Session Summary — 2026-03-01


## Problem

Clicking any toggle button (H1, H2, H3, Code, Case Sensitive, Whole Word, Regex) in the document search modal crashed the browser tab. This happened even when the search query was completely empty and the actual search logic was offloaded to a Web Worker. The crash was instant and hard — the entire Chrome tab would freeze and eventually die.

## Root Cause — Why Toggles Crashed the Browser

The crash had nothing to do with the search computation itself. The real culprit was `<ReactMarkdown>` with the `remark-gfm` plugin.

Here's what was happening step by step:

1. **The ViewerPage component renders a full markdown document.** For medium-to-large documents (1,000–10,000+ lines), `<ReactMarkdown>` has to parse the raw markdown string into an AST (Abstract Syntax Tree), walk the entire tree, and produce React elements. With `remark-gfm` enabled (which adds support for tables, strikethrough, autolinks, etc.), this parsing is even more expensive because it runs additional transformation passes over the AST.

2. **The search modal lives inside the same ViewerPage component.** When you click a toggle button like "H1" or "Case Sensitive", it calls `setOptions(...)`, which updates React state inside `DocumentSearchModal`. But here's the critical part — the modal also triggers effects that cause the **parent** `ViewerPage` to re-render.

3. **Every parent re-render re-executed `<ReactMarkdown>`.** The `<ReactMarkdown>` component was written inline in the JSX with no memoization. React has no way to know that the markdown string hasn't changed — it just sees a component in the render tree and calls it again. Every single time. That means on every toggle click, the browser was doing a full markdown parse of potentially thousands of lines of text, building a new AST, diffing it against the previous one, and producing new React elements.

4. **React Query background refetches and StrictMode compounded the problem.** React Query periodically refetches data in the background (on window focus, on interval, etc.), and React StrictMode in development double-renders components. Each of these also triggered a full `<ReactMarkdown>` re-parse. Combined with the toggle clicks, the browser was doing multiple expensive parses in rapid succession.

5. **The result: the main thread locked up and the tab crashed.** A single `<ReactMarkdown>` render on a large document can take 200–500ms+ on the main thread. Multiple re-renders stacking up with no breathing room between them meant the browser's main thread was completely blocked — no paint, no input handling, no garbage collection. The tab would freeze and Chrome would eventually kill it.

## What We Changed and Why

### Step 1: Memoize ReactMarkdown Output (Primary Fix)

**File:** `src/pages/ViewerPage.tsx`

This was the most important change. We wrapped the `<ReactMarkdown>` call in `useMemo`:

```tsx
const renderedMarkdown = useMemo(
  () => document ? (
    <ReactMarkdown components={markdownComponents} remarkPlugins={remarkPluginsConfig}>
      {document.markdown}
    </ReactMarkdown>
  ) : null,
  [document?.markdown],
);
```

Then replaced the inline `<ReactMarkdown>` in JSX with `{renderedMarkdown}`.

**What `useMemo` does here:** It caches the result of the `<ReactMarkdown>` call and only re-executes it when the dependency — `document?.markdown` — actually changes. If the parent re-renders for any other reason (modal state change, React Query refetch, StrictMode), `useMemo` returns the cached React element tree from last time. The expensive markdown parsing is completely skipped.

**Why `document?.markdown` is the right dependency:** The only time we need to re-parse the markdown is when the actual document content changes (e.g., the user navigates to a different document). Everything else — modal open/close, search options, delete button state — is irrelevant to the rendered markdown.

### Step 2: Stabilize the `onClose` Callback

**File:** `src/pages/ViewerPage.tsx`

Before:
```tsx
<DocumentSearchModal onClose={() => setIsSearchModalOpen(false)} ... />
```

After:
```tsx
const closeSearchModal = useCallback(() => setIsSearchModalOpen(false), []);
// ...
<DocumentSearchModal onClose={closeSearchModal} ... />
```

**Why this matters:** In the "before" version, `() => setIsSearchModalOpen(false)` creates a brand new function object on every render. Even though the function does the same thing every time, React sees it as a different prop because `prevFn !== nextFn` in a reference comparison. This means even with `React.memo` on the modal (Step 3), the modal would still re-render because its `onClose` prop changed.

`useCallback` with an empty dependency array (`[]`) creates the function once and returns the same reference on every subsequent render. Now when React compares props, `onClose` is the same object, and `React.memo` can correctly skip the re-render.

### Step 3: Wrap DocumentSearchModal in React.memo

**File:** `src/components/DocumentSearchModal.tsx`

Before:
```tsx
export function DocumentSearchModal({ ... }) { ... }
```

After:
```tsx
export const DocumentSearchModal = memo(function DocumentSearchModal({ ... }) { ... });
```

**What `React.memo` does:** It's a higher-order component that wraps `DocumentSearchModal` and adds a shallow prop comparison before rendering. If all props (`isOpen`, `onClose`, `documentName`, `markdown`) are the same as last render (by reference equality), React skips the entire render of the modal component.

**Why this is "defense-in-depth":** Step 1 (memoizing ReactMarkdown) is the primary fix — it prevents the expensive work even if the modal does re-render. Step 3 prevents the modal from even attempting to re-render when it doesn't need to. Together they form two layers of protection. If one fails for some reason (e.g., a new prop gets added later that isn't stabilized), the other still prevents the crash.

### Step 4: Add Worker Error Handling

**File:** `src/components/DocumentSearchModal.tsx`

Before:
```tsx
const worker = new Worker(
  new URL("../lib/searchWorker.ts", import.meta.url),
  { type: "module" },
);
```

After:
```tsx
let worker: Worker;
try {
  worker = new Worker(
    new URL("../lib/searchWorker.ts", import.meta.url),
    { type: "module" },
  );
} catch (e) {
  console.error("Failed to create search worker:", e);
  return;
}

worker.onerror = (e) => console.error("Search worker error:", e);
```

**Why:** Worker creation can fail for various reasons — the browser might not support module workers, the URL might not resolve correctly, or Content Security Policy might block it. Without a try-catch, this throws an unhandled exception that could crash the component entirely. The `onerror` handler catches runtime errors inside the worker (e.g., a syntax error in the worker script) and logs them instead of letting them propagate silently or crash the page. This is especially relevant with React StrictMode, which mounts and unmounts components twice in development — the double-mount could surface timing-related Worker issues.

### Step 5: Skip Search Scheduling on Empty Query

**File:** `src/components/DocumentSearchModal.tsx`

Before:
```tsx
useEffect(() => {
  scheduleSearch();
}, [options, scheduleSearch]);
```

After:
```tsx
useEffect(() => {
  if (queryRef.current.trim()) {
    scheduleSearch();
  }
}, [options, scheduleSearch]);
```

**Why:** When you click a toggle with an empty search query, the previous code would still call `scheduleSearch()`, which sets a 200ms debounce timer, posts a message to the Worker, and the Worker would do a bunch of setup work only to find there's nothing to search for. The `scheduleSearch` function itself does check for an empty query and returns `EMPTY_RESULT`, but only after the debounce timer fires. This change short-circuits earlier — if there's no query, don't even bother scheduling. It's a small optimization but eliminates unnecessary work and timer churn when users are just exploring the toggle buttons.

## Key Takeaway

The crash was a rendering performance problem, not a computation problem. The search was already offloaded to a Web Worker, but the real bottleneck was `<ReactMarkdown>` re-parsing thousands of lines of markdown on every React re-render. The fix was straightforward memoization — `useMemo` to cache the expensive render, `useCallback` to stabilize callback props, and `React.memo` to prevent unnecessary child re-renders. These are standard React performance patterns, but they're critical when a component's render cost is high.
