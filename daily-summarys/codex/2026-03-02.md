## Workstream 5: Memory Pressure + Crash Mitigation

Context: repeated rapid operations (large search + doc switch + filter toggles) caused browser instability/crashes.

Changes made:

- Introduced a protocolized search session model:
  - separate message contracts,
  - shared search domain models,
  - dedicated search session hook.
- Reworked worker strategy:
  - index markdown once on `set-document`,
  - execute lightweight searches with `run-search`,
  - clear indexed state on `clear-document`.
- Reduced repeated markdown payload churn between main thread and worker.
- Added safer page/result clamping behavior and effect handling to avoid stale updates and extra state churn.

Result:

- Search behavior remains feature-complete while reducing repeated high-cost operations under rapid user interaction.

### 5.1 Modularization details used to resolve search engine issues

- Introduced shared search contracts (`searchModels`) and message schema (`searchWorkerMessages`) to normalize data flow between main thread and worker.
- Added `useSearchSession` hook as the single owner of:
  - worker lifecycle,
  - document indexing triggers,
  - search scheduling state,
  - pagination reset/clamp logic.
- Reworked Search route component into composition-only orchestration (`SearchPage`) with feature modules handling controls, summary, and results.
- Outcome: lower coupling and fewer accidental rerender/worker churn paths under rapid user interactions.

## Workstream 9: Display Token View (Model-Style Tokenization)

### 9.1 Tokenization architecture
- Added a dedicated tokenization pipeline for Display Token View:
  - worker-side tokenization using `js-tiktoken/lite` with `o200k_base` ranks,
  - typed request/response contracts for worker messaging,
  - shared token document models (`lines`, `segments`, totals),
  - `useTokenSession` hook for worker lifecycle + stale-response control.
- Tokenization runs off the main thread to reduce UI blocking during large document processing.
- Cached tokenized results per active doc token to avoid unnecessary re-tokenization churn.

### 9.2 Modular token-view UI
- Introduced modular token-view components in `src/components/token/*`:
  - `TokenView` renderer,
  - row-building helper for display concerns (`tokenViewRows`),
  - token session hook (`useTokenSession`).
- Initial implementation included colorized token spans and tabbed token subviews.
- Later iterations simplified this flow (see Workstream 10).
